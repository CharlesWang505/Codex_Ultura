# Codex Compass（法典指南针）完整使用手册（适用于 1.3.0）

本文档对应 Codex Compass 1.3.0 Windows 桌面版，覆盖主界面中的页面、按钮、输入框、开关、状态提示及常见使用流程。

> **最容易被忽略的启动按钮：软件左上角的粉色闪电图标不是装饰。Codex 未运行时，点击它可启动 Codex；Codex 已运行时，点击它可重启 Codex。重新生成模型目录或更新注入模型后，也要点击它重启 Codex，使新模型菜单生效。**

## 1. 软件定位

Codex Compass（法典指南针）将三类能力整合在一个桌面程序中：

1. New API / One API 账户监控：余额、额度、调用日志、Token、成本、倍率和接口健康。
2. Codex 管理：供应商、配置文件、会话、MCP、Skills、Plugins、增强功能和脚本市场。
3. 本地热切换与代理测速：通过 `http://127.0.0.1:8787/v1` 路由不同供应商、Key 和模型，并比较 API 站点经过不同代理节点时的延迟。

“概览”和“分析”重点针对 New API、One API 及其兼容分支的管理接口开发。只有 `/v1/models`、`/v1/responses` 或 `/v1/chat/completions` 的普通 OpenAI 兼容站点，通常无法提供余额、额度、调用日志、Token、分组和倍率等后台数据。

软件中的两类配置不要混淆：

| 配置类型 | 主要认证信息 | 用途 |
| --- | --- | --- |
| 监控站点 | Cookie、User ID、后台登录信息、部分站点的 API Key | 读取账户、余额、日志、Token、分组和倍率 |
| API 供应商 | Base URL、API Key、协议和模型列表 | 发送模型请求、应用 Codex 配置、参与 8787 热切换 |

同一个 New API 站点可以只建立一个监控站点，同时为不同分组的多个 Key 建立多个 API 供应商。

软件采用本地优先设计。API Key、Cookie、登录密码、代理订阅和控制器 Secret 保存在当前 Windows 用户的应用数据目录，不会写入公开源码目录。

## 2. 安装与启动

### 2.1 便携版

直接运行 `Codex_Compass_1.3.0_x64-portable.exe`。便携版不需要安装，但运行数据仍保存在 Windows AppData，而不是 EXE 所在目录。

### 2.2 安装版

运行 `Codex_Compass_1.3.0_x64-setup.exe`，按安装向导完成安装。安装方式为当前用户安装，一般不需要管理员权限。

### 2.3 Windows 安全提示

当前公开构建未进行商业代码签名。Windows SmartScreen 可能显示“未知发布者”。应从项目的正式 GitHub Releases 下载，并使用同一版本的 `SHA256SUMS.txt` 核对文件哈希。

### 2.4 首次启动

首次启动时：

- 没有站点认证信息时，概览页显示本地示例数据。
- 8787 热切换网关默认关闭。
- 悬浮球默认关闭。
- Watcher 后台守护默认未安装。
- 自动更新默认停用，需要从 GitHub Releases 手动更新。

## 3. 全局界面

### 3.1 顶部窗口栏

- 左上角粉色闪电图标是 Codex 启动/重启按钮，不是普通品牌图标。
- Codex 未运行时点击闪电图标：启动 Codex。
- Codex 已运行时点击闪电图标：重启 Codex。
- 重新生成模型目录后点击闪电图标：让新的注入模型菜单生效。
- 点击没有反应时，到“设置 → Codex 路径与启动”检查程序路径。
- 闪电图标右侧显示 Codex Compass · 法典指南针。
- 按住空白区域可拖动窗口。
- 双击顶部空白区域可最大化或还原窗口。
- “深色 / 浅色 / 浅粉”按钮按顺序循环切换主题，并保存上次选择。
- 右上角依次为最小化、最大化/还原和关闭按钮。

### 3.2 左侧导航

导航包含：

- 概览
- 分析
- 供应商配置
- 热切换
- 会话管理
- 工具与插件
- Codex增强
- 脚本市场
- 代理测速
- 设置

### 3.3 左侧站点列表

- 每个站点显示名称和状态圆点。
- 绿色圆点表示站点具有可用认证配置。
- 灰色圆点通常表示未配置 Key、Cookie 或登录信息，软件将使用示例模式。
- 点击站点即可切换当前账户监控对象。

### 3.4 左下角账户与版本

- 显示当前中转站返回的用户名和分组。
- 未连接真实站点时显示演示账户。
- 最底部显示当前软件版本。

### 3.5 顶部站点与时间控制

除纯 Codex 页面和代理测速页外，顶部提供以下控件：

- “站点”：切换当前监控站点。
- “时间范围”：支持今天、近 24 小时、近 7 天、近 30 天和自定义时间。
- 自定义开始时间、结束时间：结束时间必须晚于开始时间。
- “应用”：应用自定义时间范围。
- “手动刷新”：立即重新读取账户、日志、模型、倍率和健康状态。
- 自动刷新倒计时：按站点配置的分钟数自动刷新。
- 最后更新：显示当前页面数据生成时间。

“今天、近 24 小时、近 7 天、近 30 天”在每次手动或自动刷新时，都会以刷新时刻重新计算结束时间。自定义时间保持用户指定的固定起止时间。

刷新期间会显示进度条：

- “日志 已加载数 / 总数”：表示调用日志分页读取进度。
- “倍率加载中 / 已就绪”：表示模型倍率接口是否完成。
- “健康度 等待中 / 完成数”：表示 API Key 健康检测进度。

## 4. 概览页面

概览页是为 New API / One API 管理接口设计的账户与调用监控面板，用于快速判断“账户还有多少额度、最近调用了什么、真实消耗是多少、接口是否正常”。

它和“供应商配置”不是同一套功能。概览读取中转站后台数据；供应商配置负责真正向模型 API 发送请求。管理接口不可用但模型接口可用时，概览可能显示“部分实时”或未知项，但这不代表该供应商不能在 Codex 中调用模型。

为了获得尽可能完整的数据，监控站点通常需要正确的 Base URL，以及站点要求的 Cookie、User ID、API Key 或登录信息。不同 New API / One API 分支开放的接口和权限不同，某些指标为空是兼容性或权限差异，软件不会用虚构数据补齐真实模式下的缺失项。

### 4.1 账户数据

- “当前余额”：读取站点账户可用余额或额度折算值。
- “额度原值”：显示站点返回的原始额度单位。
- “历史消耗”：显示账户历史累计消耗。
- “请求次数”：显示账户累计或接口返回的请求数。
- “订阅使用情况”：站点返回订阅信息时显示套餐、剩余额度和总额度。

### 4.2 使用统计（接口汇总）

- “真实消耗 Tokens”：按当前时间范围的日志汇总真实 Token 消耗。
- “实际成本”：按调用日志中的实际扣费字段汇总，不使用模型单价猜测。
- “缓存创建”：Cache Write 或缓存写入 Token。
- “缓存命中”：缓存读取 Token，并显示缓存命中率。
- “输入”：输入 Token，包含缓存读写相关字段。
- “输出”：模型输出 Token。

### 4.3 使用趋势

- 按选定时间范围显示 Tokens、实际成本、缓存命中、输入和输出曲线。
- 今天和近 24 小时使用分钟级数据并按小时标注。
- 近 7 天和近 30 天使用小时级数据并按日期标注。
- 没有调用记录时显示空状态，不生成虚假趋势。

### 4.4 模型消耗分布

- 显示不同模型的成本或 Token 占比。
- 低占比模型会合并为“其他模型”。
- 顶部显示当前范围内的实际总成本。

### 4.5 服务器可用性

- “站点状态”：检查站点基础接口是否可访问。
- “兼容模型接口”：检查 `/v1/models` 等 OpenAI 兼容接口。
- “管理接口”：检查账户、日志、令牌等后台接口。
- “刷新可用性”：只重新执行可用性检查。
- “正常”：请求成功且响应可接受。
- “高延迟”：接口可用但响应较慢。
- “异常”：接口失败或返回错误。
- “未知”：尚未检测或站点不提供该接口。

### 4.6 已用模型分组与倍率

- 只显示当前时间范围内实际出现过的模型。
- 显示模型分组、模型倍率、分组倍率、输出倍率和综合倍率。
- “编辑”允许临时校准倍率预览。
- “完成”退出本地编辑。
- 本地编辑只影响当前展示；下次刷新仍以中转站接口返回值为准。

### 4.7 令牌与调用日志

筛选控件：

- “全部模型”：按模型筛选日志。
- “全部令牌”：按默认 Key、扩展 Key或后台令牌筛选。
- “全部状态”：筛选成功或失败请求。
- 搜索框：搜索模型、令牌名称或请求 ID。
- “导出”：将当前筛选结果导出为 CSV。
- 每页 20 / 50 / 100 条：调整分页大小。

日志列说明：

- 时间：请求发生时间。
- 令牌名称：匹配到的 API Key 名称或后台令牌名称。
- 模型：实际请求模型。
- 分组：站点计费分组。
- 请求 ID：中转站请求标识。
- 状态：成功或失败。
- 输入 Tokens：输入消耗。
- 输出 Tokens：输出消耗。
- 真实总 Tokens：输入、缓存和输出的真实合计。
- 实际成本：日志实际扣费。
- 缓存：是否命中缓存。
- 用时/首字：总响应时间和首字响应时间。
- 操作：“详情”打开完整记录。

调用详情包含：

- 错误码和错误原因。
- 令牌、模型、分组、请求 ID和 IP。
- 输入、输出、缓存创建、缓存命中和真实总 Token。
- 实际成本、预扣、结算差额、最终抵扣和订阅计费信息。
- 总响应时间、首字响应时间、流式/非流式类型和输出速度。
- 推理强度、模型倍率、分组倍率、输出倍率和综合倍率。

### 4.8 数据不完整时如何判断

- 余额、额度和用户名存在，但日志为空：优先检查时间范围、Cookie 权限和日志接口权限。
- `/v1/models` 正常，余额和日志未知：说明模型兼容接口可用，但管理接口未开放或认证不完整。
- 显示“部分实时”：软件已经展示可读取的真实数据，缺失项不会用估算数据替代。
- 倍率为空：站点可能没有公开 `/api/pricing`、`/api/ratio_config` 或对应权限。
- 首字时间等于总用时：上游日志可能没有独立首字时间，或返回的是非流式请求。

## 5. 分析页面

分析页对当前站点和时间范围内的调用日志进行二次计算。

### 5.1 分析概览指标

- 平均成本 / 100 万 Tokens：衡量模型单位成本，越低越经济。
- 平均响应延迟：按日志总耗时计算。
- 平均首字用时：衡量 TTFT。
- 缓存覆盖率：缓存读取量占上下文输入的比例。
- 成本最高模型：当前范围内消耗最多的模型。
- 成本集中度：最高成本模型占总成本比例。
- 缓存节省估算：按当前均价估算缓存可能节省的金额。

### 5.2 分析图表

- 模型成本效率散点：比较模型请求数和每百万 Token 成本。
- Token 结构占比：比较输入、缓存和输出组成。
- 模型运营画像：综合展示成本效率、缓存覆盖、响应速度和首字速度。
- 模型 Token 构成：按模型拆分输入、缓存与输出。
- 时段热力分布：显示一天中各时间段的调用密度或成本。
- 缓存效率排行：比较模型缓存利用情况。
- 延迟分布与成本：观察高延迟是否同时伴随高成本。
- 首字用时分布：判断排队、冷启动和长上下文预处理问题。
- 分组成本矩阵：比较不同计费分组的模型成本。
- 模型风险评分：结合成本集中、延迟、首字和缓存效率评估风险。

### 5.3 自动建议

软件可能给出：

- 降低单模型成本集中。
- 提升缓存命中率。
- 排查高延迟调用。
- 优化首字响应。
- 继续观察当前稳定结构。

这些建议只作为运营参考，实际计费以中转站后台为准。

## 6. 供应商配置

该页面管理真正向模型上游发送请求的 API 供应商，与“站点设置”中的账户监控站点相互独立。

### 6.1 启用供应商配置

- 总开关关闭时保留所有供应商数据，但不参与 Codex 配置应用。
- 8787 热切换开启期间，供应商地址、协议和 Key 被锁定，必须先关闭热切换才能编辑。

### 6.2 供应商列表

- “添加”：创建新供应商。
- 点击供应商名称：切换右侧编辑对象。
- 列表右侧短标签显示聚合、Chat、Responses 等类型。
- “删除”：删除当前供应商；至少保留一个供应商。
- “仅保存”：位于“获取模型”和“保存并应用到 Codex”之间，只保存当前供应商配置，不写入 Codex。

### 6.3 供应商模式

- 官方模式：使用 Codex/ChatGPT 官方登录配置。
- 混合 API：保留部分官方认证，同时使用指定 API 配置。
- 纯 API：完全使用自定义 Base URL 和 API Key。
- 聚合供应商：本身不保存 Key，在多个普通供应商之间调度。

### 6.4 协议

- OpenAI Responses：适用于 `/v1/responses`。
- OpenAI Chat Completions：适用于 `/v1/chat/completions`。
- Anthropic：仅在直连 `/v1/messages` 等 Anthropic 原生接口时使用。
- Gemini：仅在直连 Gemini `generateContent` 原生接口时使用。

如果中转站已经提供 OpenAI 兼容接口，即使实际模型是 Claude 或 Gemini，也应优先选择 OpenAI 兼容协议。

### 6.5 Reasoning 方言

- 自动继承：由软件根据供应商和协议判断。
- OpenAI `reasoning_effort`。
- OpenRouter `reasoning` 对象。
- Qwen `enable_thinking`。
- 硅基流动 `enable_thinking`。
- 不发送推理参数：完全删除推理相关字段。

### 6.6 普通供应商字段

- 名称：软件内显示名称。
- Base URL：上游 API 地址，通常包含或兼容 `/v1`。
- API Key：留空时不覆盖已经安全保存的 Key；输入新值才替换。
- 默认模型：没有明确指定模型时使用。
- 测试模型：点击“测试连接”时使用；留空则使用默认模型。
- 上下文窗口：模型最大上下文，例如 `200000`。
- 自动压缩阈值：达到指定上下文量时触发压缩策略。
- 模型列表：每行一个模型，可由“获取模型”自动填充。
- 自定义 User-Agent：个别中转站要求特殊客户端标识时使用。

### 6.7 聚合供应商字段

- 成员供应商：选择已有普通 API 供应商。
- 移除失效成员：清理已经被删除的成员 ID。
- 按顺序故障切换：第一个失败后依次尝试后续成员。
- 按会话轮询：同一会话尽量保持同一成员，不同会话轮换。
- 按请求轮询：每次请求轮换成员。
- 加权轮询：按成员权重分配请求。
- 权重：只在加权轮询中生效。

### 6.8 推荐的三步操作

1. 填写 Base URL 和 API Key；普通中转站保持“纯 API”和“OpenAI Responses”即可。
2. 点击“获取模型”，然后在默认模型输入框中选择模型；获取失败时也可以手动输入。
3. 点击“保存并应用到 Codex”，软件会一次完成保存、设为默认供应商和写入 Codex 配置。

其他按钮：

- 测试连接：向测试模型发送最小请求，验证地址、Key、协议和响应；成功提示会在数秒后自动消失。
- 诊断供应商：在“诊断与清理”中检查 URL、认证、模型接口和推荐处理方式。
- 清除 Codex 供应商配置：移除 Codex Compass 写入的中转配置，恢复更接近官方状态。

### 6.9 高级维护与导入

- “读取维护状态”：读取 CCS 供应商、环境变量冲突、待导入项和 Codex 原始配置文件。
- “导入 CCS 供应商”：将检测到的兼容供应商导入当前软件。
- “移除环境变量冲突”：清除可能覆盖当前供应商的 OpenAI/Codex 环境变量。
- “读取当前 Codex 配置”：仅在软件外修改过 Codex 的 `config.toml` 或 `auth.json` 时使用，把当前文件内容回填到选中的供应商；不会自动应用其他供应商。
- 待导入“确认”：接受外部供应商导入请求。
- 待导入“忽略”：清除该导入请求。
- 原始 `config.toml`：高级用户可直接编辑并保存 Codex 配置。
- 原始 `auth.json`：包含敏感认证信息，修改前应备份。

## 7. 热切换

热切换的目标是让 Codex 始终连接一个稳定的本地地址，再由 Codex Compass 根据模型映射把请求转发到不同供应商：

```text
Codex
  → http://127.0.0.1:8787/v1
  → Codex 模型别名
  → 首选 / 备用供应商与 Key
  → 上游真实模型
```

### 7.1 开始前的供应商准备

不要先开启 8787。推荐按以下顺序准备：

1. 打开“供应商配置”，为每个上游或每个分组 Key 建立独立供应商。
2. 选择正确协议，填写 Base URL 和 API Key。
3. 点击“获取模型”，确认该 Key 能看到的模型。
4. 选择默认模型并点击“仅保存”。“仅保存”不会改写当前 Codex 配置。
5. 对其他需要参与热切换的供应商重复上述操作。
6. 再进入“热切换”配置模型目录和映射。

协议选择原则：

- 中转站提供 `/v1/responses`：选择 OpenAI Responses。
- 中转站只提供 `/v1/chat/completions`：选择 OpenAI Chat Completions。
- 直连 Anthropic `/v1/messages`：选择 Anthropic 原生协议。
- 直连 Gemini `generateContent`：选择 Gemini 原生协议。
- 中转站虽然销售 Claude、Gemini 或 Grok，但已经提供 OpenAI 兼容接口：仍选择 OpenAI 兼容协议。

### 7.2 8787 本地网关

状态包括：

- 配置已关闭。
- 配置已开启，进程未运行。
- 网关运行中。
- 网关错误。

字段说明：

- 固定供应商：自动路由关闭时使用的供应商或聚合供应商。
- 固定模型：没有模型映射命中时使用的模型。
- 默认推理强度：自动、关闭、low、medium、high、xhigh。
- 模型自动路由：根据模型别名和映射规则选择供应商。

按钮说明：

- “保存配置”：保存热切换、固定目标和自动路由设置，但不强制启动网关。
- “应用并开启”：保存当前设置，将 Codex Base URL 指向 `http://127.0.0.1:8787/v1` 并启动网关。
- “关闭热切换”：停止网关并解除供应商配置锁定。

8787 默认关闭。网关开启后，供应商的 Base URL、协议和 Key 会被锁定；需要修改时先点击“关闭热切换”。开关网关不会删除供应商或映射。

模型自动路由开启时，Codex 窗口里选择的模型优先决定路由；固定供应商和固定模型是没有映射命中时的回退目标。切换已有映射、供应商或 Reasoning 通常不需要重启 Codex。

### 7.3 模型目录生成与注入

模型目录决定 Codex 的模型菜单里能看到什么。推荐流程：

1. 勾选需要参与注入的一个或多个供应商。
2. 点击“重新生成并更新 Codex”。
3. 等待软件完成供应商模型合并、目录生成和 Codex 配置更新。
4. 检查生成的模型数量和映射规则。
5. 点击左上角闪电图标重启 Codex。
6. 在 Codex 模型菜单中选择已注入模型。

“重新生成并更新 Codex”是完整更新，不是只追加新模型。未勾选供应商的旧 Grok、Gemini、Claude 或 GPT 模型会从本次目录中清除，避免 Codex 保留已经无法路由的旧条目。

1.2.10 已修复多供应商目录只返回当前单一供应商的问题。实机验证中，ChatGPT 供应商的 12 个模型与 Claude-Kiro 的 18 个模型可合并为 30 个注入模型。实际数量取决于各供应商返回的模型列表。

只有重新生成或改变注入白名单后必须重启 Codex；在现有目录和映射中进行热切换通常不用重启。左上角闪电图标的作用就是启动/重启 Codex，而不是重新生成映射。

### 7.4 模型映射规则

每条规则包含：

- Codex 模型 / 别名：Codex 侧看到或请求的模型名。
- 上游真实模型：实际发送给供应商的模型名。
- 首选供应商：默认使用的供应商/Key。
- 备用供应商顺序：首选失败后按顺序尝试。
- Reasoning 覆盖：继承、关闭、low、medium、high 或 xhigh。

操作按钮：

- 新增规则：添加空白映射。
- 自动扫描：从全部供应商获取模型并生成或补充映射。
- 保存规则：验证并保存映射。
- 删除：删除当前规则。
- 备用供应商“上移 / 下移”：调整故障切换顺序。
- “移除”：从备用列表删除供应商。
- “添加备用供应商”：加入其他可用供应商。

别名和真实模型可以不同。例如 Codex 中显示 `claude-sonnet`，网关可以把它映射为上游实际接受的 `claude-sonnet-4-20250514`。同一个模型名在多个供应商中出现时，只需要选择一次首选供应商，再安排备用顺序。

首选供应商请求失败时，网关按备用列表顺序尝试。备用供应商必须确实支持映射中的上游真实模型，否则切换后仍会失败。

人工修改别名或映射后先点击“保存规则”。再次自动扫描或重新生成时，软件会尽量保留有效的人工规则；如果取消了对应供应商或上游模型已经消失，相关规则可能需要重新确认。

常见校验错误：

- 模型别名为空。
- 上游模型为空。
- 首选供应商不存在。
- 备用供应商包含首选供应商。
- 备用供应商失效或重复。
- 模型别名重复。

### 7.5 Reasoning 转换

界面统一使用 `自动 / 关闭 / low / medium / high / xhigh`。网关再按供应商配置转换成 OpenAI `reasoning_effort`、OpenRouter `reasoning` 对象、Qwen/硅基流动 `enable_thinking`，或完全移除推理参数。

单模型映射中的 Reasoning 覆盖优先于供应商方言和全局默认值。上游不支持某个强度时，应改为该上游实际支持的值，或选择“关闭”。

### 7.6 悬浮切换

- 开启开关后立即显示悬浮球，不需要额外点击保存。
- 保存悬浮设置：保存其他悬浮相关设置。
- 恢复默认位置：把悬浮球移动到默认屏幕位置。

悬浮球操作：

- 单击：打开或关闭快速切换面板。
- 拖动：调整悬浮球位置并保存。
- 右键：打开 Codex Compass 主窗口。

快速面板可以：

- 选择供应商/Key。
- 输入或选择模型。
- 选择 Reasoning 强度。
- 应用配置并开启 8787。
- 关闭或开启网关。
- 打开主程序。

悬浮面板只能切换已经保存的供应商和模型。需要编辑 Base URL、Key、协议或重新注入模型时，仍应回到主窗口。

### 7.7 一套完整的热切换操作

1. 关闭 8787。
2. 在“供应商配置”中获取模型并“仅保存”全部供应商。
3. 进入“热切换”，勾选要注入的供应商。
4. 点击“重新生成并更新 Codex”。
5. 检查并保存模型映射。
6. 点击“应用并开启”。
7. 点击左上角闪电图标重启 Codex。
8. 在 Codex 中选择模型并发送一次请求。
9. 如首选供应商失败，检查请求日志和备用映射是否实际支持同一个上游模型。

## 12. 代理测速

代理测速用于比较“同一个 API Base URL 经过不同 Clash/Mihomo 节点时的真实可达性和延迟”。它不是普通网页测速，也不是节点带宽测试。

软件对每个“代理节点 × Base URL”组合分别发起测试，因此同一个节点可能访问 A 站很快、访问 B 站超时。代理测速不会切换 Clash 当前节点，也不会修改系统代理。

### 12.1 代理测速概况

- 控制器：显示外部 Clash/Mihomo 或内置引擎连接状态。
- 已选机场：参与测速的订阅数量。
- 可测速路线：当前筛选后可用的代理节点和本地直连数量。
- 本轮结果：成功率或尚未开始状态。

### 12.2 本机控制器

连接方式：

- 命名管道：Windows Clash Verge 常用，默认尝试 `\\.\pipe\verge-mihomo`。
- HTTP：连接 Mihomo External Controller 地址。

控件：

- 控制器地址：填写命名管道或 HTTP 地址。
- Secret：控制器认证 Secret；留空可保持已保存值。
- 清除已保存 Secret：明确删除旧 Secret。
- 自动发现：扫描常见 Clash/Mihomo 控制器。
- 连接节点：使用发现结果读取节点。
- “请先停用内置引擎”：内置引擎运行时不能同时接管外部控制器。

如果使用 HTTP External Controller，应填写 Clash/Mihomo 实际监听地址；控制器启用了 Secret 时必须同时填写。命名管道和 HTTP 控制器只需选择一种。

### 12.3 内置测试引擎

- 安装并启用：首次从 MetaCubeX 官方 GitHub 下载独立 Mihomo 核心。
- 启动：启动已经安装的内置核心。
- 停用：停止内置核心并恢复外部控制器配置。
- 内置核心只监听本地地址，不修改系统代理。
- 安装前至少选择一个机场订阅。

内置引擎适合没有运行 Clash 客户端，或不希望让测速依赖当前 Clash 配置的情况。它只服务于 Codex Compass 的测试流程，不会接管浏览器和其他应用流量。

### 12.4 代理订阅

- 订阅名称：为机场设置便于识别的名称。
- 订阅 URL：支持 Clash YAML、Base64、VMess、SS、Trojan、VLESS、Hysteria2、TUIC 等常见格式。
- 导入：保存订阅地址并解析节点名称。
- 更新订阅：重新读取节点名称和流量信息。
- 移除订阅：只删除 Codex Compass 保存的订阅配置，不修改 Clash。
- 选择框：决定该机场是否参与测速。
- 仅选择此机场：取消其他机场并只选择当前项。
- 全选：选择全部机场。
- 清空：取消全部机场。
- 仅本地：清空机场并保留本地直连测试。

订阅卡可能显示已用流量、总流量、到期时间和识别到的节点数。

订阅 URL 通常包含访问凭证，应当像密码一样保护。软件保存订阅地址和解析结果，但公开截图、Issue 或日志前仍应隐藏完整 URL。

### 12.5 Base URL 目标

- 加入当前站点：把当前监控站点加入测速目标。
- 加入全部已配置站点：批量加入所有有效 HTTP/HTTPS 站点。
- 自定义：新增空白目标。
- 启用开关：决定该目标是否参与本轮测速。
- 目标名称：用于矩阵表头。
- Base URL：要测试的实际访问地址。
- 删除目标：移除该测速目标。

公开版初始目标为 `example.com` 示例地址，应替换成自己实际使用的上游或中转站地址。

Base URL 可填写站点根地址或 API 根地址，例如 `https://api.example.com` 或 `https://api.example.com/v1`。测速不发送模型请求，不需要填写 API Key；应优先填写日常真实使用的地址，而不是通用测速网站。

### 12.6 测试参数

- 超时：3、5、8、12 或 20 秒。
- 并发：2、4、6、8、12 或 16 个同时执行的测速任务。
- 仅所选机场：只测试已选订阅匹配到的节点。
- 包含本地直连：把当前网络直连作为独立路线。
- 路线搜索：按路线名称或协议过滤。
- 状态筛选：全部路线、可用路线、本轮失败。
- 排序：平均延迟、成功目标数或路线名称。
- 保存测速设置：保存控制器、订阅选择、目标、超时、并发和筛选偏好。

建议日常使用 4～8 并发。并发过高可能触发代理节点或 API 站点的连接限制，反而产生大量失败。首次比较时建议开启“包含本地直连”，用当前网络直连结果作为基准。

### 12.7 开始、停止和导出

- 开始测速：按“路线 × Base URL”执行实时延迟测试。
- 停止：停止创建新任务；已经发出的少量请求会自然结束。
- 导出 CSV：导出当前延迟矩阵。

开始前至少需要一个启用的 Base URL，并且需要选择代理订阅或启用本地直连。测速过程中可以用路线搜索和状态筛选缩小结果范围。

### 12.8 延迟矩阵

- 每行代表一个代理节点或本地直连。
- 每列代表一个 Base URL。
- 单元格显示毫秒、失败、超时、等待或未测试。
- 平均列显示该路线成功目标的平均延迟。
- 成功目标数显示该路线成功访问的目标数量。
- 点击目标表头第一次按该目标延迟升序，第二次降序，第三次恢复平均延迟排序。
- “最佳路线”摘要显示最低平均延迟和成功数量。
- 本地直连会标注“不经过 Mihomo”，用于判断代理是否真的改善了访问质量。

不要只看“平均延迟”。某条路线可能在一个站点很快，但其他站点全部失败。选择节点时应依次查看：

1. 成功目标数是否覆盖自己需要的全部站点。
2. 点击最重要的 Base URL 表头，查看该站点的单列排序。
3. 再比较平均延迟和本地直连基准。

点击某个 Base URL 表头会循环“延迟正序 → 延迟倒序 → 恢复平均延迟排序”。

### 12.9 推荐测速流程

1. 先连接外部控制器，或选择订阅后启动内置引擎。
2. 导入并勾选需要比较的机场。
3. 加入当前站点、全部站点或自定义 Base URL。
4. 开启“包含本地直连”。
5. 超时先使用 8 秒，并发先使用 4 或 6。
6. 点击“开始测速”。
7. 先按成功目标数排除不可用路线。
8. 点击最重要的 Base URL 表头，找到该站最快节点。
9. 必要时导出 CSV，留存不同时间段的结果。

## 13. 设置页面

设置页同时包含站点设置和 Codex 设置。

### 13.1 关闭窗口行为

- 每次关闭时询问：点击关闭按钮后显示选择弹窗。
- 关闭到系统托盘：隐藏主窗口，自动刷新和后台检测继续运行。
- 直接退出软件：结束刷新、测速、网关和后台任务。

关闭确认弹窗：

- 最小化到系统托盘。
- 关闭软件。
- 取消。
- “记住本次选择”：以后直接执行，可在设置中恢复。

### 13.2 站点设置

- 名称：站点在左侧列表中的名称。
- Base URL：中转站主页或管理接口地址，必须以 `http://` 或 `https://` 开头。
- API Key / 用户访问令牌：默认认证 Key。
- 显示/隐藏敏感信息：切换 Key、Cookie 和密码可见状态。
- 检测更多 / 收起更多：展开或收起高级认证配置。

多 Key 配置：

- 导入令牌：从站点令牌列表导入能够读取到的完整 Key。
- 添加 Key：新增扩展 Key。
- 默认：默认 Key 行。
- 令牌名称：用于健康卡和调用日志匹配。
- API Key：完整 Key 或用户访问令牌。
- 启用：决定该 Key 是否参与健康检查。
- 删除扩展 Key：删除扩展 Key，不影响默认 Key。

高级认证：

- 用户 ID：New API 的后台用户 ID，通常是纯数字。
- Cookie / Session：管理接口登录 Cookie。
- New API 登录账号：账号或邮箱。
- New API 登录密码。
- Cookie 失效时自动登录：接口鉴权失败时尝试 `/api/user/login` 并更新 Cookie。
- 登录获取 Cookie：立即执行登录。

站点类型：

- 自动识别。
- New API / One API。
- OpenAI 兼容。

其他操作：

- 自动刷新（分钟）：最小 1 分钟。
- 添加站点：创建新站点。
- 保存配置：验证并保存敏感配置。
- 删除：删除当前站点；至少保留一个站点。

认证信息至少配置 API Key、Cookie 或账号密码自动登录中的一种，否则页面使用示例数据。

### 13.3 Codex 路径与启动

- Codex 应用路径：Codex 或 ChatGPT 桌面程序可执行文件路径。
- 增强补丁模式：启动时启用客户端增强注入。
- 中转模式：使用供应商或网关配置启动。
- 额外启动参数：按空格拆分并传给 Codex。
- 保存：保存启动设置。

### 13.4 后台守护 Watcher

状态：

- 后台守护未安装。
- 启动注册需要修复。
- 已停止。
- 待启动。
- 运行中。

按钮：

- 检测：读取安装、注册和运行状态。
- 安装：为当前 Codex Compass 主程序注册后台守护入口。
- 启用：允许 Watcher 运行。
- 停用：关闭 Watcher，但保留安装入口。

Watcher 使用当前 Codex Compass EXE 的隐藏参数启动，不应弹出命令行黑窗。

### 13.5 危险操作

“重置 Codex 功能设置”只重置 Codex 供应商、增强和相关功能设置，不删除中转站账户监控配置。执行前仍建议备份。

### 13.6 更新与诊断

- 检查更新：当前版本没有配置 Codex Compass 自有可信更新源，因此只返回停用说明。
- 下载安装：只有未来配置可信更新源并检测到版本后才可用。
- 读取日志：读取最近的 Codex Compass/Codex 诊断日志。
- 复制诊断：生成并复制脱敏诊断摘要。
- 日志文本框：显示最近日志。
- 诊断文本框：显示可复制的诊断报告。

当前更新方式是从项目 GitHub Releases 手动下载新版。

### 13.7 关于

显示：

- Codex Compass、中文名“法典指南针”和版本。
- 中转站监控、Codex 兼容核心和本地优先存储说明。
- 敏感配置不会进入普通日志和诊断摘要的提醒。

## 16. 数据保存与备份

Windows 下应用数据目录由 Tauri 标识 `chat.ai-api.relay-meter` 决定。虽然软件已改名为 Codex Compass，但保留该内部标识以继续读取旧版 API Detect 与 Codex_Ultura 数据。

典型内容包括：

- `sensitive/sites.json`：站点、Key、Cookie 和登录配置。
- `sensitive/proxy-latency.json`：代理订阅、控制器和测速配置。
- `settings/app-preferences.json`：关闭窗口行为等非敏感偏好。
- `codex/settings.json`：Codex Compass 的 Codex 功能设置。
- `codex/logs`：诊断日志。
- `codex/scripts`：用户和市场脚本。
- `codex/backups` 或相关备份目录：会话、配置和迁移备份。

升级或重装前应备份整个 `chat.ai-api.relay-meter` 数据目录。不要把该目录上传到 GitHub、网盘公开链接或问题截图中。

## 17. 推荐使用流程

### 17.1 只监控中转站

1. 打开“设置”。
2. 确认该站属于 New API、One API 或兼容分支；普通 OpenAI 兼容接口可能只能显示部分数据。
3. 填写站点名称和 Base URL。
4. 填写该站点要求的 API Key、Cookie、User ID 或登录账号密码。
5. 点击“登录获取 Cookie”或“保存配置”。
6. 返回“概览”，选择时间范围后点击“手动刷新”。
7. 在“服务器可用性”中区分模型接口和管理接口状态。

### 17.2 Codex 使用单个中转供应商

1. 打开“供应商配置”。
2. 新建纯 API 供应商。
3. 填写协议、Base URL、Key 和默认模型。
4. 点击“测试连接”和“获取模型”。
5. 点击“保存并应用到 Codex”。

### 17.3 一个中转站使用多个分组 Key

1. 账户监控只创建一个站点。
2. 在“供应商配置”中为每个分组 Key 建立独立供应商。
3. 分别获取各 Key 的可用模型，并逐个点击“仅保存”。
4. 在“热切换”中勾选这些供应商。
5. 点击“重新生成并更新 Codex”。
6. 为同名模型设置首选 Key 和备用 Key 顺序。
7. 保存规则后点击“应用并开启”。
8. 点击左上角闪电图标重启 Codex。

### 17.4 开启悬浮快速切换

1. 先完成供应商和模型映射。
2. 打开“热切换”。
3. 开启悬浮球。
4. 单击悬浮球打开面板。
5. 选择供应商、模型和推理强度。
6. 点击“应用并开启”。

### 17.5 比较不同代理节点访问中转站

1. 打开“代理测速”并连接 Clash/Mihomo 控制器。
2. 导入订阅并选择机场。
3. 加入实际使用的一个或多个 Base URL。
4. 开启“包含本地直连”，并发选择 4～8。
5. 开始测速。
6. 先看成功目标数，再点击重点 Base URL 表头进行单列排序。

## 18. 常见问题

### 18.1 页面显示示例数据

原因通常是没有配置认证信息。检查 Base URL、API Key、Cookie、User ID 和自动登录设置，然后保存并手动刷新。

### 18.2 账户数据正常但日志为空

检查时间范围、Cookie 权限、日志接口权限和该时间段是否真的产生调用。可在“设置 → 接口来源”查看接口状态。

### 18.3 获取模型失败

检查 Base URL 是否包含正确的 `/v1` 路径、Key 是否有模型列表权限，以及供应商协议是否正确。部分中转站不开放 `/v1/models`，需要手动填写模型列表。

### 18.4 8787 开启后供应商无法编辑

这是保护机制。先在“热切换”中点击“关闭热切换”，再返回供应商配置编辑地址、协议和 Key。

### 18.5 Codex 切换后仍使用旧供应商

确认 Codex Base URL 已指向 `127.0.0.1:8787/v1`，检查网关状态，并确认模型映射命中。模型自动路由开启时以 Codex 当前选择的模型为准；固定供应商只是未命中映射时的回退。官方模式与网关模式切换时重新应用一次配置。

### 18.6 悬浮球看不到

在“热切换”重新关闭并开启悬浮球，然后点击“恢复默认位置”。悬浮球可能被保存到已断开的显示器坐标。

### 18.7 悬浮球能看到但面板打不开

单击时不要拖动鼠标。移动超过少量像素会被识别为拖动。仍无法打开时重启 Codex Compass，并确认没有安全软件阻止置顶透明窗口。

### 18.8 Watcher 定时弹出黑色窗口

当前实现使用无控制台后台启动。若仍出现黑窗，先点击“检测”，再执行“安装”和“启用”以修复旧注册；同时删除旧 CodexPlusPlus Watcher 自启动项。

### 18.9 代理节点匹配不到

先在 Clash Verge/Mihomo 更新对应机场订阅，确认控制器节点名称与 Codex Compass 解析的名称一致；也可以关闭“仅所选机场”查看控制器全部节点。

### 18.10 自动更新为什么不可用

Codex Compass 尚未配置独立可信更新源。自动下载被主动停用，避免误装 CodexPlusPlus 或未知安装包。请从正式 GitHub Releases 手动更新。

### 18.11 Windows 提示未知发布者

公开构建未签名。只从正式仓库下载，并核对 SHA-256。未知发布者不等于文件损坏，但无法证明发布者身份。

### 18.12 注入后 Codex 仍显示上一次的模型

重新进入“热切换”，只勾选本次需要的供应商，再点击“重新生成并更新 Codex”。生成完成后点击左上角闪电图标重启 Codex。重新生成会完整替换注入集合，未勾选供应商的旧模型应被清除。

### 18.13 勾选 Claude 供应商但模型没有进入 Codex

先回到“供应商配置”，确认该供应商已经“获取模型”并“仅保存”。如果中转站提供 OpenAI 兼容接口，应选择 OpenAI Responses 或 Chat Completions，而不是因为模型叫 Claude 就选择 Anthropic 原生协议。然后重新勾选供应商、生成目录并重启 Codex。

### 18.14 `-compact` 模型为什么没有显示

先确认上游模型列表或生成后的目录中确实存在该模型。Codex 模型菜单不是简单显示所有文本名称，目录白名单、模型元数据和当前勾选的供应商都会影响结果。重新生成目录后必须重启 Codex；如果上游没有返回 `-compact`，可在模型映射中使用 Codex 兼容别名映射到真实上游模型。

### 18.15 代理测速全部失败

检查控制器是否已连接、订阅是否被勾选、至少一个 Base URL 是否启用，以及目标地址是否能被当前网络访问。HTTP 控制器启用了 Secret 时必须填写正确 Secret。也可以先启用“包含本地直连”：如果直连也失败，优先检查 Base URL 和本机网络，而不是代理节点。

## 19. 隐私与安全注意事项

- 不要在截图中展示完整 Key、Cookie、请求 ID、余额、真实站点域名和订阅地址。
- 不要把 AppData 数据目录提交到 GitHub。
- 原始 `auth.json` 和 `config.toml` 可能包含敏感信息。
- 诊断报告已尽量脱敏，公开前仍应人工检查。
- 脚本具有修改 Codex 页面或配置的能力，只安装可信来源脚本。
- 代理订阅包含访问凭证，应视为密码保存。
- 修改或删除会话前保留软件生成的备份。

## 20. 浏览器预览与桌面版的区别

运行 `npm run dev` 只用于前端界面预览：

- 不读取真实 Key。
- 不读取真实 Codex 配置。
- 不执行 Tauri 命令。
- 不执行真实代理测速。
- 不启动 8787 网关。

完整功能必须运行 Codex Compass 桌面 EXE 或使用 `npm run dev:tauri`。
